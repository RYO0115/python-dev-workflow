# 拡張性を損ねるハードコードの禁止 — 詳細と具体例

「ユーザーが自分でデータを自由に登録・拡張できる」ことを中核に据えるアプリでは、
ユーザーが登録しうる値の集合をコード側で決め打ちしてはいけない。

## 禁止するパターン

- ユーザーが登録するデータの内容を、コード内の固定リスト/辞書で限定すること。
- 品詞・カテゴリ・言語など、ユーザー入力や AI 生成で増減しうる値を、
  コード内の固定 enum 的リストで網羅しようとすること。

### アンチパターンの実例

過去に約 55 個の不規則動詞活用形を直接記述した辞書（`_IRREGULAR_VERB_FORMS`）を持っていた。
リストに無い単語をユーザーが登録すると機能が正しく動かなくなった。
→ 汎用ライブラリ（`lemminflect`）への置き換えで解消した。

## 実装前の自問

新しいロジックを書くとき「この値が想定外だったら動かなくなるか？」を自問する。
**Yes** の場合、以下の代替を優先順に検討する。

1. **既存データ（DB）から動的に算出する。**
   例: 登録済みデータ同士の類似度計算、入力履歴からのサジェスト生成。
2. **AI 連携で動的に生成する。**
3. **汎用の外部ライブラリに委譲する。**（形態素解析・NLP・活用形生成など）

## 対象外（ハードコードしてよい）

**技術的に閉じている値**は対象外。以下はユーザーの語彙内容に関係しないため通常通り書いてよい。

- HTTP ステータスコード
- DB のカラム名・テーブル名
- 設定キー名
- 固定の UI 文言・ラベル
- 正規表現パターン
- プロトコル定数・列挙が仕様上確定しているもの

## 判断に迷ったら

固定リストを書く前に一度立ち止まり、上記 1〜3 の代替手段を検討する。
「今は網羅できるが将来ユーザーが増やせる」種類の値なら、動的算出・生成側に倒す。
