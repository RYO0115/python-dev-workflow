# テストの詳細方針

## いつテストを追加するか

機能を**追加・修正・削除**したら毎回。純粋な内部リファクタ（外部の振る舞いが変わらない）だけは対象外。判断に迷ったら追加する。

## テストの種類の使い分け

| 検証したいもの | 種類 | 例のツール |
|---------------|------|-----------|
| ブラウザでの見た目・操作（HTMX/JS の挙動含む） | UI テスト | Playwright |
| ロジック単体を決定的に検証（ランダム性・日時条件など UI 越しでは検証しづらいもの） | ユニットテスト | pytest |

推奨ディレクトリ構成:

```
tests/
├── unit/     # サービス層など、決定的なロジックの単体テスト
└── ui/       # ブラウザ操作・表示のテスト
```

## 完了前の確認

- `uv run pytest` を実行して**全テスト green** を確認してから完了報告する。
- テストが落ちている場合は、出力を添えて落ちている事実を正直に報告する。

## 既存テストとの関係

- 変更が既存テストの対象範囲内なら、新規ファイルを作らず既存テストを修正する。
- テスト一覧ドキュメントがあるプロジェクトでは、追加・変更したテストケースを追記する。
