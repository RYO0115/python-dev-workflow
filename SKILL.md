# Python 開発ワークフロー共通ルール

Python プロジェクトを実装・拡張・レビューする際に従う共通ルール。
プロジェクト固有の CLAUDE.md がある場合はそちらを優先し、本スキルはその土台として使う。

## 使いどころ

- Python アプリ / ライブラリの新規実装・機能追加・修正・削除
- 依存関係・仮想環境・バージョンの管理（uv / pyproject.toml / bump-my-version）
- アーキテクチャ設計やリファクタの方針決め
- テスト・ドキュメントの追従漏れチェック

## 実装前チェックリスト

1. 変更対象の周辺コードを読み、既存の構成・命名・コメント密度に合わせる。
2. これから書くロジックが「ユーザー入力やAI生成で増減しうる値」を固定リストで決め打ちしていないか自問する（→ [ハードコード禁止原則](#拡張性を損ねるハードコードの禁止必須)）。
3. 変更が外部の振る舞いを変えるなら、対応するテストとドキュメント更新もセットで計画する。

## 1. 開発環境・依存管理

- **Python 3.12+ / uv** を標準とする。依存の追加は `uv add`、実行は `uv run` を使う。
- 依存やロックの手動編集は避け、`uv` コマンド経由で更新して `uv.lock` を同期する。
- 環境変数は `.env` で管理し、`.env.example` にキー設定例を必ず用意する（値はダミー）。

## 2. バージョン管理

- バージョンは `pyproject.toml` の `version` フィールドで**一元管理**する。
- 変更後は `bump-my-version` でインクリメントする（`pyproject.toml` 更新・git commit・git tag を自動化）。

| 変更の種類 | コマンド | 例 |
|-----------|----------|----|
| 破壊的変更・アーキテクチャ変更 | `uv run bump-my-version bump major` | 0.1.0 → 1.0.0 |
| 機能追加 | `uv run bump-my-version bump minor` | 0.1.0 → 0.2.0 |
| バグ修正・細かい調整 | `uv run bump-my-version bump patch` | 0.1.0 → 0.1.1 |

> minor を上げると patch はリセット、major を上げると minor・patch はリセットされる。

## 3. アーキテクチャ・ディレクトリ設計

- `core/`（共通基盤: DB接続・ベースモデル・例外など）と `features/`（機能別モジュール）に分ける。

```
src/<package>/
├── core/          # 共通基盤
└── features/      # 機能別モジュール（feature ごとにサブディレクトリ）
```

- **依存方向は一方向のみ**。`core ← features/*` の向きを守り、逆方向・循環参照は禁止。
- features 同士が依存する場合も一方向に限定し、依存グラフを設計書に明記する。
- コードの重複を避け、共通処理は `core/` に集約する。

詳細は [references/architecture.md](references/architecture.md)。

## 4. コメント規則

- ソースファイル冒頭に、そのファイルの概要コメントを必ず記載する。
- 重要なクラス・関数・変数にコメントを付ける。
- 「何をするか(what)」ではなく「**なぜそうするか(why)**」を説明する。

## 5. テスト追加・更新ルール（必須）

機能を追加・修正・削除したら、**毎回必ず**テストを追加または更新する。

- ブラウザでの見た目・操作を検証すべき場合: UI テスト（例: Playwright）。
- ロジック単体を決定的に検証したい場合（ランダム性・日時条件など）: ユニットテスト。
- 既存テストの対象範囲内の変更なら、新規ファイルではなく既存テストの修正でよい。
- 完了報告の前に `uv run pytest` を実行し、**全テストが green** であることを確認する。

外部の振る舞いが変わらない純粋な内部リファクタは対象外でよいが、迷ったらテストを追加する。
詳細は [references/testing.md](references/testing.md)。

## 6. ドキュメント同期

- 機能追加・変更・削除をしたら、設計書・機能一覧・API 仕様などの関連ドキュメントを同じコミット内で更新する。
- 各ドキュメント冒頭の「最終更新」日付・バージョンも更新する。

## 7. 拡張性を損ねるハードコードの禁止（必須）

ユーザーが自由にデータを登録・拡張できることを前提とするアプリでは、以下を**禁止**する。

- ユーザーが登録するデータの内容を、コード内の固定リスト/辞書で限定すること。
- 将来ユーザー入力やAI生成で増減しうる値の集合を、固定 enum 的リストで網羅しようとすること。

**実装前の自問**: 「この値が想定外だったら動かなくなるか？」— Yes なら以下を優先する。

1. 既存データ（DB）から動的に算出する。
2. AI 連携で動的に生成する。
3. 汎用ライブラリ（形態素解析・NLP 等）に処理を委譲する。

ただし**技術的に閉じている値**（HTTP ステータス・DBカラム名・設定キー・固定UI文言・正規表現など、ユーザーの語彙内容に関係しない値）は対象外で、通常通りハードコードしてよい。

詳細と具体例は [references/no-hardcode-principle.md](references/no-hardcode-principle.md)。
